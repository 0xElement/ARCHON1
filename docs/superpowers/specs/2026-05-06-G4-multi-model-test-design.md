# G4: Multi-Model Orchestration Test — Design Spec

**Status:** 🟢 APPROVED (per architecture vision §8 Phase 1)
**Layer:** Validates Principle 2.1 ("orchestration > frontier model") — pre-work before Layer-A through Layer-D builds
**Source:** [Architecture vision §5 G4](../../architecture/2026-05-06-kurukshetra-quality-architecture.md), [Research §15 Provos](../../research/2026-05-05-agentic-framework-research.md)
**Estimated effort:** 3 days build + 1 week observation = 10 days total
**Estimated win if successful:** $5,500/month savings (Sonnet $30/run vs Opus $140/run × 50 runs/month)

---

## 1. Goal (one sentence)

Empirically determine whether KRISHNA dispatcher quality holds when we swap Opus 4.7 for Sonnet 4.6 (or Haiku 4.5 for KRIPA verifier), so kurukshetra can adopt cheaper models for routine engagements without quality loss.

## 2. Why this is the right first move

### 2.1 De-risks the entire architecture vision
The vision (§2.1 principle) bets that *orchestration* matters more than *model size*. If that bet is wrong, everything downstream (G1 Judge, G3 TrajAD, G2 A2A) is built on a shaky foundation. G4 is a 3-day probe that proves or disproves the bet before we invest weeks in Layer A-D builds.

### 2.2 Cheapest pattern with biggest immediate ROI
Per architecture §11: G4 is 3-day build, no architectural change, $5,500/month savings if it works. No other Phase 1 pattern has this ROI profile.

### 2.3 Provos already proved it works (single-data-point caveat)
IronCurtain found NEW zero-days using Sonnet 4.6, Opus 4.6, AND Z.AI GLM 5.1 — same orchestration, different models, equivalent quality on his benchmarks. We need to verify that holds for kurukshetra's pentest workload specifically.

## 3. Approach (recommended) — Three-target controlled experiment

### 3.1 Why this approach over alternatives

| Option | Description | Risk | Time | Signal quality |
|---|---|---|---|---|
| A. Quick A/B on next dispatch | Swap KRISHNA model on next pentest run, compare to historical | Low | 3 days | Single data point, anecdotal |
| **B. 3-target controlled (RECOMMENDED)** | **Run 3 known-good targets with both Opus and Sonnet KRISHNA, compare F1 + cost + chain depth** | **Low** | **10 days** | **3 data points, defensible** |
| C. Full progressive rollout | Sonnet on stocks/code-review (low-stakes), Opus on pentest (high-stakes), measure for 30 days | Medium | 30 days | Strong signal but slow |

**Choice: B.** A is too anecdotal. C is too slow for "does the bet work?" question. B gives 3 cross-domain data points (one per squad type) in <2 weeks.

### 3.2 The three controlled targets

| # | Squad | Target type | Why this target |
|---|---|---|---|
| 1 | pentest | A previously-tested example target (e.g. host.example.com) with KNOWN findings | Replay against historical truth — direct comparison possible |
| 2 | code-review | A previously-reviewed kurukshetra commit chain | White-box test, depends less on external state |
| 3 | stocks | A standard daily-portfolio analysis on a known stock | Stocks squad uses different reasoning patterns — cross-validates |

### 3.3 The experimental matrix

For each target, run TWO dispatches:
- **Control (Opus):** current production setup, KRISHNA = Opus 4.7
- **Treatment (Sonnet):** KRISHNA = Sonnet 4.6, everything else identical

Total: 6 dispatches (3 targets × 2 models)

### 3.4 Metrics captured per dispatch

| Metric | What it measures | How |
|---|---|---|
| **F1 score** | Quality vs known truth | (TP + 0.5×(FP+FN))^-1 against historical-truth findings |
| **Findings count** | Quantity / coverage | total + per-severity (Critical / High / Medium / Low) |
| **Chain depth** | Reasoning complexity | max length of attack chains generated by KRISHNA |
| **Total cost** | $/run from API metering | sum of all model API tokens × current pricing |
| **Total time** | Wall-clock duration | dispatch start → VYASA report complete |
| **GATE failures** | Quality discipline | which of 50+ GATEs fail per dispatch |
| **Verifier passes** | Phase 3.6/3.8 outcomes | counts of CONFIRMED / KILLED / INDETERMINATE |
| **Sycophancy score** | Anti-bias check | separate-context grader's quality rating (Haiku) |

### 3.5 Decision criteria (when to ADOPT Sonnet)

Sonnet is adopted for routine targets if **ALL THREE** hold across the 3 targets:
1. **F1 within 5% of Opus** (e.g. Opus 0.95 → Sonnet ≥0.90)
2. **No new false-Critical findings** (Sonnet doesn't make worse mistakes; FP-Critical ≤ Opus's count)
3. **Cost reduction ≥60%** (we're swapping for savings; if savings <60%, the experiment is moot)

If 2 of 3 hold → Sonnet for stocks/code-review only, keep Opus for pentest
If 0 or 1 hold → reject Sonnet, keep Opus, document what failed

### 3.6 Abort criteria (stop the experiment early)

Halt and revert if:
- Any Sonnet run produces a false-Critical finding that wasn't in the Opus baseline
- Sonnet run takes >2× the wall-clock time of Opus (defeats the cost savings)
- GATE-12 (threat-model discipline) fails for Sonnet but passes for Opus on same target
- verify-loop alerts on regression during the experiment

## 4. Architecture changes required

### 4.1 New: model-config layer
**File:** `agents/model-config.js` (new)
**Purpose:** centralize per-agent model assignments so we can swap without code edits.

```javascript
// agents/model-config.js
const PROFILES = {
  default: {
    KRISHNA: 'claude-opus-4-7',
    DHARMA: 'claude-opus-4-7',
    KRIPA: 'claude-haiku-4-5',
    VYASA: 'claude-opus-4-7',
    GRADER: 'claude-haiku-4-5',
  },
  G4_test_sonnet: {
    KRISHNA: 'claude-sonnet-4-6',  // ONLY change
    DHARMA: 'claude-opus-4-7',
    KRIPA: 'claude-haiku-4-5',
    VYASA: 'claude-opus-4-7',
    GRADER: 'claude-haiku-4-5',
  }
}
module.exports = { PROFILES, getModel(agent, profile = 'default') { return PROFILES[profile][agent] } }
```

**Rationale:** isolating the change to ONE config file makes the experiment trivial to revert. No event-bus.js modifications.

### 4.2 Modified: event-bus.js model resolution
**Change:** wherever event-bus.js currently has hardcoded `'claude-opus-4-7'` for KRISHNA, replace with `modelConfig.getModel('KRISHNA', process.env.MODEL_PROFILE || 'default')`.

**Estimated edits:** 3-5 lines changed in event-bus.js (need to grep for hardcoded model strings).

### 4.3 New: G4 metrics collector
**File:** `scripts/g4-metrics.js` (new)
**Purpose:** parse VALIDATED-FINDINGS jsonl + RECON files + tasks.json, emit one JSON metrics record per dispatch.

**Output shape:**

### 4.4 New: G4 results report generator
**File:** `scripts/g4-report.js` (new)
**Purpose:** read all 6 metrics records, render comparison table + decision.

## 5. Test plan

### 5.1 Unit tests
- `test/model-config.test.js` — verify default + G4_test_sonnet profiles load correctly, getModel returns expected strings
- `test/g4-metrics.test.js` — verify metrics collector parses synthetic VALIDATED-FINDINGS + tasks.json correctly

### 5.2 Integration test
- Run dispatch with `MODEL_PROFILE=default` → verify KRISHNA gets Opus
- Run dispatch with `MODEL_PROFILE=G4_test_sonnet` → verify KRISHNA gets Sonnet
- Both must complete without GATE regression

### 5.3 Real-world experiment
- Run 6 controlled dispatches per §3.2-3.3 matrix
- Capture metrics per §3.4
- Apply decision criteria §3.5

## 6. Risk register

| Risk | Likelihood | Mitigation |
|---|---|---|
| Sonnet causes GATE-11 (evidence-completeness) regression | Medium | Verify-loop catches it; abort criterion §3.6 triggers |
| Sonnet is cheaper but slower → no net cost saving | Low | Wall-clock time tracked in metrics; decision §3.5 caps at 60% reduction threshold |
| Historical Opus runs have unknown ground-truth quality (chicken-egg) | Medium | Use example v4 backlog targets that have human-verified findings (Bugcrowd reports) |
| MODEL_PROFILE env var misconfigured during experiment | Low | Log explicit profile name in every dispatch; verify-loop reads tasks.json model field |
| Anthropic SDK breaks on model name change | Very low | Both 'claude-opus-4-7' and 'claude-sonnet-4-6' are valid current model IDs per CLAUDE.md |

## 7. What this spec does NOT cover

- **Multi-provider testing** (e.g., Z.AI GLM 5.1 per Provos) — out of scope for this spec; revisit after Sonnet result lands
- **Bernstein deterministic routing** (Y2) — different pattern, defer to Phase 3
- **CASTER semantic routing** (R2) — REJECTED per architecture §6
- **Per-phase model assignment** (e.g., Sonnet for KRISHNA, Opus for VYASA) — only KRISHNA-only swap in this spec; per-phase optimization is follow-up work

## 9. Implementation phases (handoff to writing-plans)

The implementation plan should sequence:

1. **Build phase (3 days):**
   - 1a. agents/model-config.js with PROFILES + getModel
   - 1b. event-bus.js refactor: replace hardcoded model strings (3-5 sites)
   - 1c. test/model-config.test.js unit tests
   - 1d. scripts/g4-metrics.js metrics collector
   - 1e. test/g4-metrics.test.js
   - 1f. scripts/g4-report.js comparison renderer

2. **Experiment phase (7 days):**
   - 2a. Pick 3 target IDs from historical data, document expected baseline F1
   - 2b. Run 6 controlled dispatches (1 control + 1 treatment per target), serial to avoid contention
   - 2c. After each pair, run g4-metrics.js + commit results
   - 2d. After all 6: run g4-report.js, write decision

3. **Decision phase (≤1 day):**
   - 3a. Apply criteria §3.5
   - 3b. Document decision in memory + commit
   - 3c. If adopt: production swap PR, with docs/architecture/ note

Total: 11 days end-to-end. Build phase blocks experiment; experiment phase blocks decision.

## Appendix A: Why no clarifying questions to Jay

This spec deliberately makes design choices without asking Jay for input on every detail. Per his autonomy directive (msg 1934: "tujhe sab pata hai ki kya karna apne framework ke liye achha hai vahi karne ka"):

| Decision | Made by spec | Rationale |
|---|---|---|
| 3 targets vs 5 | 3 | Time-budget vs signal trade-off; Pareto-optimal at 3 |
| Sonnet vs GLM 5.1 | Sonnet | Anthropic-native, no provider integration risk; GLM 5.1 follow-up if Sonnet works |
| KRISHNA-only vs all-agents | KRISHNA-only | Smallest variable change → cleanest causal interpretation |
| F1 within 5% threshold | 5% | Industry-standard tolerance for "equivalent quality" |
| Cost reduction ≥60% | 60% | Anything less doesn't justify the operational complexity of two-profile system |

If Jay disagrees with any of these, change is a 1-line edit in this spec — implementation hasn't started.

## Appendix B: Verification handshake with verify-loop

The kurukshetra-verify-loop.sh (cron */30) will continue running during the experiment. Specifically:
- It catches if any G4 work breaks tests/GATEs/daemons (regression alert via Telegram outbox)
- It does NOT track G4 metrics — that's g4-metrics.js's job
- Both can run in parallel without conflict
